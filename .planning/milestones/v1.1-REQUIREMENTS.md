# Requirements Archive: v1.1 Cloud-First / No-Docker Workspace Tooling

**Archived:** 2026-04-04  
**Defined:** 2026-04-03  
**Outcome:** All v1 requirements completed

## v1 Requirements

### Workspace Defaults

- [x] **WSDF-01**: The recommended workspace start path uses cloud-first defaults and does not require Docker for normal backend or app development.
- [x] **WSDF-02**: Docker-backed local modes remain available only as explicit opt-in paths for narrow local-infra cases.
- [x] **WSDF-03**: Workspace doctor and preflight checks explain clearly when Docker is optional versus actually required.

### Database Validation

- [x] **DBVL-01**: The docs define a safe remote-first migration validation path using isolated Supabase branches or disposable database targets.
- [x] **DBVL-02**: Backend schema validation guidance does not assume local `supabase start` or `supabase db reset` as the default milestone verification path.
- [x] **DBVL-03**: The preferred remote validation path keeps production and shared persistent databases out of the blast radius.

### Script And Profile Alignment

- [x] **SCPT-01**: Root workspace scripts and `Makefile` targets align with the cloud-first default instead of silently routing developers into Docker-dependent flows.
- [x] **SCPT-02**: Screenalytics-specific local infra startup is isolated so unrelated backend or app work is not blocked by Docker availability.
- [x] **SCPT-03**: Shared profiles and env docs reflect one canonical no-Docker preferred path.

### Operational Adoption

- [x] **ADPT-01**: At least one real verification path proves milestone checks can be run in this workspace without Docker.
- [x] **ADPT-02**: Handoff and status artifacts point to the same cloud-first validation contract as the scripts and docs.
- [x] **ADPT-03**: Remaining Docker-only cases are explicitly documented as fallback or special-case behavior rather than hidden defaults.
